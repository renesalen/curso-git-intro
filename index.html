Flujo de trabajo de GIT

Working Directory

Hizimos nuestro 'git init' para indicar que estamos iniciando nuestro trabajo. Los Archivos (html, css, js, images/) aún no tienen no tienen seguimiento

¿Cómo Sabemos qué archivos no tienen seguimiento?

con: 'git status -s'nos va pintar si los archivos se modificaron, se eliminaron o si se crearon

después de enter el '-s' nos va pintar ¿?

Una vez que nosotros tenemos identificado los archivos que queremos guardar, GIT va tomar arreglos provisorios para que a futuro si hacemos cambios podamos volver a esta version

Staging Area, o Area Temporal

Se trata del AREA LOCAL. Para hacer un seguimiento debemos enviar el archivo a esta area local con: 'git add' indicando los archivos a ser agregados: 'git add index.html'. ( si no hay mensajes en el terminal, es señal que va funcionando bien)

Cuando hacemos otra vez: 'git status -s' en la lista 'index.html' aparece con una A adelante (añadido al Area Temporal)

Local Repo o Repositorio Local

Del Area local podemos enviar al REPOSITORIO LOCAL con: 'git commit'. El commit es una FOTOGRAFIA que se guarda en el REPOSITORIO, es una especie de albun de imagenes. El COMMIT es el que nos permite el viaje en el tiempo

Para tomar una foto (y enviarlo al Repositorio) hacemos: 'git commit -m "agregamos el index.html". Enter y aparece que se nos agregó un archivo. (la -m es para agregar un mensaje)

Una vez el archivo agregado al repositorio, el nombre del archivo en la lista VScode pierde su color verde y la letra A que lo sigue

a los Proyectos ses llaman Repositorios en Git
